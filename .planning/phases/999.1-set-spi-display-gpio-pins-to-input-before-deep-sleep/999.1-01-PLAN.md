<objective>
Set the SPI and display-control GPIO pins to high-impedance INPUT mode before `esp_deep_sleep_start()` in the battery path of `hibernate()`, to eliminate leakage current through the e-paper display's ESD protection diodes during deep sleep.

The Seeed_GFX `epaper.sleep()` call (already issued by `update()`) only sends a software sleep command over SPI — it does NOT change GPIO directions. After it, SCLK (GPIO8), MOSI (GPIO9), DC (GPIO10), CS (GPIO44), CS1 (GPIO41), and RST (GPIO38) all remain driven OUTPUTs. GPIO8/9/10/38/41/44 are digital-only pins on the ESP32-S3 (NOT RTC-capable), so `rtc_gpio_isolate()` is not applicable — `SPI.end()` (for the muxed bus pins) plus `pinMode(pin, INPUT)` (for the control pins) is the correct approach.

Purpose: Reduce deep-sleep battery drain (continuation of the Phase 10 GPIO-isolation work that already isolated GPIO1/GPIO6). Impact may be small or zero if the T133A01 already tri-states its own SPI inputs on sleep — so measurement is part of the deliverable, not an afterthought.
Output: A modified `hibernate()` battery path and a documented before/after deep-sleep current measurement.
</objective>

    Insert exactly this block:
    ```cpp
    // Tri-state SPI/display control pins before deep sleep to eliminate leakage
    // through the e-paper protection diodes. epaper.sleep() (sent by update())
    // only issues a software command — it does NOT change GPIO directions.
    // GPIO8/9/10/38/41/44 are digital-only on ESP32-S3 (NOT RTC-capable), so
    // rtc_gpio_isolate() does not apply; use SPI.end() + pinMode(INPUT) instead.
    // Do NOT use gpio_reset_pin() — community reports it can block deep-sleep entry.
    SPI.end();                  // releases GPIO8 (SCLK) and GPIO9 (MOSI) from the SPI peripheral
    pinMode(DC_PIN,  INPUT);    // GPIO10
    pinMode(CS_PIN,  INPUT);    // GPIO44
    pinMode(CS1_PIN, INPUT);    // GPIO41
    pinMode(RST_PIN, INPUT);    // GPIO38
    ```
