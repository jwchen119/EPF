## Exact Diff Applied to hibernate()

Inserted after `rtc_gpio_isolate(GPIO_NUM_6);` and before `fs_deinit();`:

```cpp
// Tri-state SPI/display control pins before deep sleep to eliminate leakage
// through the e-paper protection diodes. epaper.sleep() (sent by update())
// only issues a software command — it does NOT change GPIO directions.
// GPIO8/9/10/38/41/44 are digital-only on ESP32-S3 (NOT RTC-capable), so
// rtc_gpio_isolate() does not apply; use SPI.end() + pinMode(INPUT) instead.
// Do NOT use gpio_reset_pin() — community reports it can block deep-sleep entry.
SPI.end();                  // releases GPIO8 (SCLK) and GPIO9 (MOSI) from the SPI peripheral
pinMode(DC_PIN,  INPUT);    // GPIO10
pinMode(CS_PIN,  INPUT);    // GPIO44
pinMode(CS1_PIN, INPUT);    // GPIO41
pinMode(RST_PIN, INPUT);    // GPIO38
```

## Decisions Made

- `SPI.end()` placed after `rtc_gpio_isolate()` calls and before `fs_deinit()` — correct sequencing ensures display has already received software sleep command via `update()` before SPI bus is released
- Named macros (DC_PIN, CS_PIN, CS1_PIN, RST_PIN from epdif.h) used instead of raw numbers — avoids magic constants and matches the rest of the codebase
- `gpio_reset_pin()` explicitly not used — community reports it can block deep-sleep entry on ESP32-S3
- Cleanup applied to `hibernate()` battery path only — USB path calls `ESP.restart()` and reinitializes everything, so cleanup there would be unnecessary and potentially harmful
