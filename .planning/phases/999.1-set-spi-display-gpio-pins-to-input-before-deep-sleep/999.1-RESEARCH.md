# Phase 999.1: Set SPI/Display GPIO Pins to INPUT Before Deep Sleep - Research

**Researched:** 2026-06-28
**Domain:** ESP32-S3 GPIO management, deep sleep leakage, Seeed_GFX EPaper driver internals
**Confidence:** HIGH

---

## Summary

This phase addresses leakage current through the e-paper display's ESD protection diodes during deep sleep.
The SPI and display control pins (DC GPIO10, CS GPIO44, CS1 GPIO41, RST GPIO38, SCLK GPIO8, MOSI GPIO9) are
driven as OUTPUTs by the Seeed_GFX library until `esp_deep_sleep_start()` is called. When the chip enters
deep sleep, the digital peripheral power domain (APB_CLK domain) is powered down. The GPIO output drivers
turn off, but the pads may float or be weakly driven by internal pull-ups rather than their programmed output
values. If the e-paper display's SPI pins have ESD/protection diodes steering to VDD, and VDD is still
applied to the display, leakage can flow through the diode and through the now-floating or high-impedance
ESP32-S3 GPIO pad.

**Critical finding from source inspection:** `epaper.sleep()` in Seeed_GFX sends only a software sleep
command (`0x07`, `0xa5`) to the display IC over SPI. It does NOT reconfigure GPIO pin directions. After
`epaper.sleep()`, all SPI/control pins remain in OUTPUT mode. There is no automatic tri-state.

**Critical finding on ESP32-S3 pin capabilities:** GPIO8, GPIO9, GPIO10, GPIO38, GPIO41, GPIO44 are
**digital-only pins** on the ESP32-S3 — they are NOT RTC-capable. `rtc_gpio_isolate()` cannot be used
on them. Deep sleep automatically powers down the digital GPIO peripheral; pin states after power-down
depend on pad hold configuration.

**Primary recommendation:** Before `esp_deep_sleep_start()`, call `pinMode(pin, INPUT)` (Arduino API) on
each of the 5-6 SPI/control pins. This configures them as high-impedance inputs before the digital domain
powers off, eliminating the GPIO output driver as a source of current into the display protection diodes.
Measure actual deep-sleep current before and after to determine real-world impact — the e-paper display may
already pull these lines via its own internal resistors regardless.

## Architecture Patterns

### Current State After `epaper.sleep()` + Before `esp_deep_sleep_start()`

```
GPIO8  (SCLK) : OUTPUT — driven by SPI peripheral
GPIO9  (MOSI) : OUTPUT — driven by SPI peripheral
GPIO10 (DC)   : OUTPUT — driven by Seeed_GFX (HIGH or LOW from last command)
GPIO38 (RST)  : OUTPUT — driven HIGH (display running)
GPIO41 (CS1)  : OUTPUT — driven HIGH (deselected)
GPIO44 (CS)   : OUTPUT — driven HIGH (deselected)
GPIO4  (BUSY) : INPUT  — already input, no change needed
```

When deep sleep starts, the APB_CLK domain powers down. Without pad hold (`gpio_deep_sleep_hold_en`),
these OUTPUT pins lose their driven state. The pads then sit in an indeterminate state. Internal pull-ups
or pull-downs may apply depending on default pad configuration. If the display's SPI lines are pulled
to 3.3V by the display's own circuitry, and the ESP32-S3 pad is in a floating or weakly pulled-down
state, a small current flows through the ESD protection diode.

### Recommended: Set Pins to INPUT Before Deep Sleep

```cpp
// In hibernate(), BEFORE esp_deep_sleep_start():
// Set SPI/display control pins to INPUT (high-impedance) to prevent
// leakage through e-paper protection diodes during deep sleep.
// Note: These are digital-only pins — rtc_gpio_isolate() cannot be used.
// Note: epaper.sleep() only sends a software command; it does NOT tri-state these pins.
pinMode(DC_PIN,   INPUT);  // GPIO10
pinMode(CS_PIN,   INPUT);  // GPIO44
pinMode(CS1_PIN,  INPUT);  // GPIO41
pinMode(RST_PIN,  INPUT);  // GPIO38
// SCLK and MOSI are SPI peripheral pins — release after SPI.end()
SPI.end();  // Releases GPIO8 (SCLK) and GPIO9 (MOSI) from the SPI peripheral
```

**Important:** `SPI.end()` must be called before `pinMode()` on SCLK/MOSI, since those pins are muxed to the
SPI peripheral. Without `SPI.end()`, `pinMode(GPIO8, INPUT)` may be overridden by the SPI driver on next
SPI transaction. After `SPI.end()`, calling `pinMode(GPIO8, INPUT)` and `pinMode(GPIO9, INPUT)` is safe.

However, calling `SPI.end()` means `epaper.begin()` on the next wakeup will need to re-initialize the SPI
peripheral — which it already does via `EPD_WAKEUP()` → `EPD_INIT()` inside `EPaper::wake()`.

### Alternative: gpio_set_direction (ESP-IDF)

```cpp
gpio_set_direction((gpio_num_t)DC_PIN,  GPIO_MODE_INPUT);
gpio_set_direction((gpio_num_t)CS_PIN,  GPIO_MODE_INPUT);
gpio_set_direction((gpio_num_t)CS1_PIN, GPIO_MODE_INPUT);
gpio_set_direction((gpio_num_t)RST_PIN, GPIO_MODE_INPUT);
```

This is equivalent to `pinMode(pin, INPUT)` but uses the ESP-IDF GPIO driver directly.

### Placement in Existing `hibernate()` Function

The changes belong in the battery path of `hibernate()`, immediately before `esp_deep_sleep_start()`:

```cpp
void hibernate(int sleepDuration = 0)
{
    // ... existing USB path (no change needed — ESP.restart() reinitializes everything) ...

    // Battery power path:
    WiFi.disconnect(true);
    WiFi.mode(WIFI_OFF);

    pinMode(CS_PIN,  INPUT);        // GPIO44
    pinMode(CS1_PIN, INPUT);        // GPIO41
    pinMode(RST_PIN, INPUT);        // GPIO38
    // GPIO8 (SCLK) and GPIO9 (MOSI) are released by SPI.end()

    fs_deinit();

}
```

### Measurement Approach (as recommended in phase description)

The phase description explicitly says to measure impact first. The procedure:

1. Flash existing firmware with a serial current-measurement probe on the battery line.
2. Record deep-sleep current baseline (target: ~20 µA chip, ~100-200 µA board-level).
3. Flash firmware with the `pinMode(INPUT)` additions.
4. Record deep-sleep current after change.
5. If delta is negligible (< 10 µA), note that the e-paper `Sleep()` driver call effectively
   tri-states the display's SPI interface, making the GPIO change a low-impact improvement.
6. If delta is measurable (> 50 µA), document in commit and SUMMARY.

### Anti-Patterns to Avoid

- **`gpio_reset_pin()`:** At least one community report states this prevents deep sleep on ESP32. Avoid. Use
  `pinMode(INPUT)` instead, which only sets direction without the full pad reset gpio_reset_pin performs
  (enabling pullups and disabling output).
- **Calling `rtc_gpio_isolate()` on non-RTC pins:** GPIO8, GPIO9, GPIO10, GPIO38, GPIO41, GPIO44 are NOT
  RTC-capable on ESP32-S3. `rtc_gpio_isolate()` on these pins will return an error or silently have no
  effect. Check via `rtc_gpio_is_valid_gpio()` if in doubt.
- **Skipping `SPI.end()` before `pinMode` on GPIO8/GPIO9:** The SPI peripheral mux takes priority. Without
  releasing the peripheral, setting GPIO8/GPIO9 to INPUT via Arduino is unreliable.
- **Calling `SPI.end()` before `epaper.sleep()`:** `epaper.sleep()` sends a command over SPI. Must call
  `epaper.sleep()` (or confirm it was called already in `update()`) before `SPI.end()`.
- **Omitting `TFT_ENABLE` / GPIO43 consideration:** The `EPaper_Board_Pins_Setups.h` file defines
  `TFT_ENABLE 43` for the EE02 board. This may be a display power enable pin. If driven HIGH, it may supply
  power to the display panel during sleep, providing a return path for protection diode leakage. Investigate
  whether GPIO43 needs to be set LOW or INPUT as well.

## Key Source Finding: `epaper.sleep()` Does NOT Tri-State Pins

From direct inspection of the installed Seeed_GFX library at
`/Users/lennart/Documents/Arduino/libraries/Seeed_GFX/`:

**`Extensions/EPaper.cpp`, line 223-229:**
```cpp
void EPaper::sleep()
{
    if (_sleep)
        return;
    EPD_SLEEP();   // ← sends 0x07 command over SPI; no GPIO direction changes
    _sleep = true;
}
```

**`TFT_Drivers/T133A01_Defines.h`, line 166-172:**
```cpp
#define EPD_SLEEP()         \
    do                      \
    {                       \
        writecommanddata(0x07, Sleep_V, sizeof(Sleep_V));  // Sleep_V = {0xa5}
        delay(1);           \
        CHECK_BUSY();       \
    } while (0)
```

Conclusion: `EPD_SLEEP()` is a pure SPI command to the display IC. No Arduino `pinMode()`, no
`gpio_set_direction()`, no `SPI.end()` is called. After `epaper.sleep()`, the SCLK, MOSI, DC, CS, CS1,
and RST pins remain in OUTPUT mode.

**`update()` calls `sleep()` automatically** (EPaper.cpp line 66), so after `processImageData()` returns,
all display operations including the driver sleep command have already been sent. By the time `hibernate()`
is called, the display is in its low-power mode but the SPI bus is still active.

## Common Pitfalls

### Pitfall 1: Calling `gpio_reset_pin()` Instead of `pinMode(INPUT)`

**What goes wrong:** Deep sleep may fail to enter or current increases unexpectedly.
**Why it happens:** `gpio_reset_pin()` re-enables internal pull-ups and performs additional pad
configuration that may interact badly with the pre-sleep state machine.
**How to avoid:** Use `pinMode(pin, INPUT)` (Arduino) or `gpio_set_direction(pin, GPIO_MODE_INPUT)`
(ESP-IDF). These only change direction.
**Warning signs:** Serial log shows `gpio_reset_pin` called but `esp_deep_sleep_start()` never returns,
or sleep current is anomalously high (> 1 mA).

### Pitfall 2: Forgetting `SPI.end()` for GPIO8/GPIO9

**What goes wrong:** SCLK (GPIO8) and MOSI (GPIO9) stay driven by the SPI peripheral even after
`pinMode(GPIO8, INPUT)` — the peripheral mux takes priority over GPIO matrix mode.
**Why it happens:** `pinMode()` sets the GPIO matrix, but the SPI peripheral has its own IO mux override.
**How to avoid:** Always call `SPI.end()` before `pinMode()` on SPI bus pins.
**Warning signs:** After adding `pinMode(GPIO8, INPUT)` and `pinMode(GPIO9, INPUT)`, current measurement
shows no change — the peripheral is still driving the pins.

### Pitfall 3: Calling `SPI.end()` Before `epaper.sleep()`

**What goes wrong:** If `SPI.end()` is called while the display is still active (before the software sleep
command is sent), the `EPD_SLEEP()` macro's `writecommanddata()` call will fail silently. The display
remains in active mode and draws full refresh current.
**Why it happens:** `EPD_SLEEP()` calls `spi.transfer()` internally; after `SPI.end()`, the `spi` object
is uninitialized.
**How to avoid:** Confirm `epaper.sleep()` has been called (it is called by `update()` at line 66 of
EPaper.cpp). Only call `SPI.end()` in `hibernate()`, which is called after `processImageData()` returns.
**Warning signs:** Display shows residual content from partial update; "display never fully sleeps" current
observed.

### Pitfall 4: The Change May Be a No-Op

**What goes wrong:** Measurement shows no change in deep-sleep current.
**Why it happens:** The T133A01 display may tri-state its own SPI interface input lines when in sleep mode
(command `0x07`), eliminating the protection-diode leakage path from the display side. The change is still
correct engineering practice but may have immeasurable impact.
**How to avoid:** Measure before and after. If no change is observed, document that the display driver's
sleep command already handles the leakage path, and keep the `pinMode(INPUT)` calls as defensive code.
**Warning signs:** Pre/post current measurements within noise floor (< 5 µA difference).

### Pitfall 5: TFT_ENABLE (GPIO43) Not Considered

**What goes wrong:** Even with SPI pins set to INPUT, the display panel remains powered through GPIO43
(`TFT_ENABLE` in EPaper_Board_Pins_Setups.h for EE02), sustaining a leakage path.
**Why it happens:** `TFT_ENABLE` is defined for EE02 in `EPaper_Board_Pins_Setups.h` but its function
(power enable vs. reset gating) is not documented in the firmware's `epdif.h`.
**How to avoid:** Verify what GPIO43 drives on the EE02 schematic. If it powers the display panel,
consider setting it LOW (not INPUT — if it's a power rail, you want to drive it LOW, not float it).
**Warning signs:** Deep-sleep current remains high even after SPI pin INPUT configuration.

## Open Questions

1. **Does the T133A01 tri-state its SPI input lines when sleep command 0x07 is received?**
   - What we know: `EPD_SLEEP()` sends command `0x07` with data `{0xa5}`. The T133A01 datasheet would
     specify whether the display IC tri-states its SPI input pads after receiving this command.
   - What's unclear: The T133A01 datasheet is not publicly available in English; Seeed has not published it.
   - Recommendation: Measure deep-sleep current before/after the firmware change to empirically determine
     whether the display IC already handles this. If the pre-change current is already at expected baseline
     (~100-200 µA board-level), the firmware change is still good defensive code but the leakage from SPI
     pins specifically may be negligible.

2. **What does TFT_ENABLE (GPIO43) do on the EE02 board?**
   - What we know: `EPaper_Board_Pins_Setups.h` defines `TFT_ENABLE 43` for EE02 variants (EE04, EE05 also
     use it). It is not mentioned in `epdif.h` or the main `.ino` file, suggesting Seeed_GFX manages it
     internally.
   - What's unclear: Whether GPIO43 is a power-enable line (in which case driving it LOW before sleep is
     important) or a chip-select variant.
   - Recommendation: Verify with EE02 schematic. If it is a power-enable active-HIGH, drive GPIO43 LOW
     before sleep. If it is not currently used by the application, this is likely a non-issue.

3. **Should `SPI.end()` be called if `begin()` re-initializes via `EPD_WAKEUP()`?**
   - What we know: `EPaper::begin(wake=false)` calls `EPD_INIT()` which re-sends all SPI init commands.
     `begin(wake=true)` calls `initFromSleep()`. The SPI peripheral is initialized by `TFT_eSPI::init()`.
   - What's unclear: Whether calling `SPI.end()` in `hibernate()` and then `epaper.begin()` on the next
     wakeup will cause a double-initialization issue or whether `TFT_eSPI::init()` transparently handles
     the SPI re-setup.
   - Recommendation: Test empirically. If `SPI.end()` causes issues on wakeup, use only
     `gpio_set_direction()` for SCLK/MOSI instead, accepting that the SPI peripheral may hold the pin
     states until deep sleep actually powers down the digital domain.
